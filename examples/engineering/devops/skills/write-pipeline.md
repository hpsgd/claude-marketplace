**Stage ordering rationale:** Lint runs first — it takes under 30 seconds and catches the cheapest class of errors. Build and unit tests run in parallel after lint (both `needs: lint`) since neither depends on the other's output. Integration tests wait for unit tests to pass — no point running real database tests if units are broken. Security scan runs last before deploy to avoid blocking fast feedback with a slower scan. Deploy runs only on main after everything passes.

**Caching:** Cache key is `node-${{ hashFiles('package-lock.json') }}`. The cache hits on every run where dependencies haven't changed, which should be ~90% of pushes. Restore key `node-` provides a partial cache hit when the lockfile has changed — better than a cold install. If cache hit rate drops below 80%, the lockfile path is likely wrong.

**Fast path time budget:** lint (30s) + unit tests (parallel with build, ~60s) = ~90 seconds. Well inside the 10-minute budget.

**Rollback plan:** Deploy stage uses the artifact built in the build job (same SHA). To roll back, re-run the deploy job from a previous successful workflow run in the GitHub Actions UI, or run `./scripts/deploy.sh <previous-sha>` manually. The deploy script should support targeted SHA deployment.
