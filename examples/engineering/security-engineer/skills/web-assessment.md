**Pre-assessment checklist (logged before starting any active testing):**

- Target URL and scope: which URLs and functionality are in scope

- Testing restrictions: any specific tests or hours excluded

Opening `[!IMPORTANT]` callout: "This skill requires explicit written authorisation for the target application. Active testing sends requests to the target — this is not passive. Never test production systems without written authorisation that explicitly includes production."

**Step 2 — Authentication assessment:**

Credential testing: default credentials for the platform, account enumeration (do login error messages differ for "user doesn't exist" vs "wrong password"?), lockout policy and bypass attempts.

Password policy: minimum length enforcement, server-side validation (not just client-side), weak password acceptance.

MFA: availability, enforcement for privileged accounts, TOTP replay protection, SMS SIM-swap exposure.

Password reset flow: token randomness, single-use tokens, short-lived tokens, predictable reset questions.

**Step 3 — Session management:**

Token quality: 128+ bits of entropy, randomness, not in URL parameters.

Cookie attributes: `HttpOnly` (prevents JavaScript access), `Secure` (HTTPS only), `SameSite` (CSRF mitigation), appropriate expiry.

Session lifecycle: invalidated on logout, invalidated after password change, absolute timeout.

**Step 4 — Authorisation:**

IDOR: change `/api/orders/12345` to `/api/orders/12346` while authenticated as a different user — access to another user's data = IDOR.

Vertical escalation: capture admin request, replay as standard user — access to admin function = privilege escalation.

Forced browsing: directly access unlinked URLs — protected by auth or just hidden by obscurity?

**Step 5 — Input validation with specific payloads:**

SQL injection: single quote `'`, `' OR '1'='1`, `'; DROP TABLE` (non-production only). `sqlmap` in safe mode for systematic testing.

XSS reflected/stored/DOM-based: `<script>alert(1)</script>`, `innerHTML`, `document.write`, `eval()` with user-controlled data.

SSRF: `http://169.254.169.254/latest/meta-data/` (AWS metadata), `http://localhost/`, internal hostnames.

XXE: `<!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]>`.

Command injection: `; id`, `| id`, `&& id`.

**Step 6 — Security headers:**

```bash
curl -I -s "[URL]" | grep -i -E "strict-transport|content-security|x-frame|x-content-type|referrer-policy|permissions-policy"
```

| Header | Expected value | Risk if absent |
|---|---|---|
| Strict-Transport-Security | `max-age=31536000; includeSubDomains` | Downgrade attack |
| Content-Security-Policy | Restrictive policy | XSS escalation |
| X-Frame-Options | `DENY` or `SAMEORIGIN` | Clickjacking |
| X-Content-Type-Options | `nosniff` | MIME sniffing |
| Referrer-Policy | `strict-origin-when-cross-origin` | Data leakage in referrer |

**Immediate critical finding reporting:**

Rules: "Report critical findings immediately — don't wait for the full assessment to complete."

**Step 7 — API security:**

Exposed documentation (`/api/docs`, `/swagger`, `/openapi.json`), same auth/authz controls as web UI, mass assignment (API accepts unexpected fields), rate limiting, verbose error messages revealing stack traces or schema.
