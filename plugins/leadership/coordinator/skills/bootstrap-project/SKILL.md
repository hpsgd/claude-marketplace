# Bootstrap Project

Scaffold the foundational documentation and governance structure for $ARGUMENTS. This skill generates the starting-point artifacts that every project needs — architecture decisions, quality gates, security policy, and operational docs — tailored to the project type. Every generated file needs team review; this is a launchpad, not a finished product.

## Step 1: Gather Context

Before creating anything, establish the project parameters. Ask the user (or infer from codebase if bootstrapping an existing project):

```markdown
### Project Parameters

| Parameter | Value |
|---|---|
| **Project name** | [name] |
| **Type** | [API / web app / mobile app / library / data pipeline / CLI tool] |
| **Tech stack** | [languages, frameworks, infrastructure] |
| **Team size** | [number of engineers] |
| **Regulatory requirements** | [GDPR, SOC 2, HIPAA, PCI-DSS, none] |
| **AI features planned?** | [Yes/No] |
| **Deployment target** | [cloud provider, on-prem, hybrid] |
```

Scan the codebase with `Glob` and `Grep` to detect existing docs, CI config, and project structure. Don't overwrite existing artifacts — extend or skip.

**Output:** Completed parameters table. List of existing artifacts found (if any).

## Step 2: Create Directory Structure

Create the docs directory layout appropriate to the project type:

```
docs/
├── adr/                    # Architecture Decision Records
├── architecture/           # System design (arc42)
├── api/                    # API design documents (API projects only)
├── specs/                  # Specification documents
├── okrs/                   # OKR documents
├── test-strategy/          # Test strategy and quality gates
├── governance/             # Risk register, compliance, policies
├── runbooks/               # Operational runbooks
└── performance/            # Performance budgets, capacity plans
```

Skip directories that don't apply:
- `api/` — only for projects with API surfaces
- `governance/` — only for regulated projects or team size > 3
- `runbooks/` — only for deployed services (not libraries or CLI tools)

Use `Bash` with `mkdir -p` to create the structure. Check existing directories first — never clobber.

**Output:** List of directories created.

## Step 3: Generate Foundational Documents

Generate documents based on project type. Every generated file must include the marker comment:

```

```

### Always generated (all project types)

## Step 4: Generate Quality Gates

Create pre-merge and pre-release checklists as concrete, verifiable items:

```markdown
<!-- Generated by bootstrap-project. Review and customize. -->
# Quality Gates

## Pre-Merge Checklist

- [ ] All tests pass (unit, integration, contract)
- [ ] Code review approved by at least one reviewer
- [ ] No new lint warnings or suppressions without ADR
- [ ] Test coverage does not decrease
- [ ] API changes have updated OpenAPI spec (if applicable)
- [ ] Database migrations are reversible
- [ ] No secrets in code (checked by pre-commit hook)
- [ ] Acceptance criteria from the spec are met
- [ ] Documentation updated (if user-facing change)

## Pre-Release Checklist

- [ ] All pre-merge checks pass on release branch
- [ ] CHANGELOG.md updated with release notes
- [ ] Performance budget verified — no regressions
- [ ] Security scan clean (dependencies + SAST)
- [ ] Runbook updated for new operational concerns
- [ ] Rollback procedure tested or documented
- [ ] Stakeholder sign-off on release scope
```

Write this to `docs/test-strategy/quality-gates.md`.

**Output:** Quality gates file created.

## Rules

- **Don't generate everything.** Only create artifacts relevant to the project type. A library doesn't need runbooks. A data pipeline doesn't need API design guidelines. Match artifacts to context.
- **Never overwrite existing files.** If an artifact already exists, skip it and note it in the summary. Existing docs represent decisions already made.
- **Every file gets the marker comment.** `<!-- Generated by bootstrap-project. Review and customize. -->` at the top of every generated file. This signals that the content is a starting point.
- **Use agent templates.** Every generated document must follow the format defined by the responsible agent's skill (e.g., ADRs follow `/architect:write-adr`, test strategy follows `/qa-lead:test-strategy`).
- **CHANGELOG.md uses [Keep a Changelog](https://keepachangelog.com/) format.** Sections: Added, Changed, Deprecated, Removed, Fixed, Security. Start with `## [Unreleased]`.
- **SECURITY.md follows [GitHub conventions](https://docs.github.com/en/code-security/getting-started/adding-a-security-policy-to-your-repository).** Include supported versions, reporting process, and expected response time.
- **This is a starting point.** Say it in the summary. Say it in the marker comments. Every doc needs team review before it becomes authoritative.

## Output Format

Summary table listing all created files, their purpose, and what action the team needs to take on each. Follow with numbered next steps.
