When writing or reviewing any text output (documentation, commit messages, PR descriptions, comments, user-facing copy), apply these guidelines:

## Quick reference
- Active voice, second person ("you")
- Imperative mood for commands and commit messages
- One idea per sentence, short paragraphs
- Lead with the point — no preamble
- Plain language — no jargon without context
- Concrete examples over abstract explanations
- "Use" not "utilize", "build" not "implement", "help" not "facilitate"
- No filler: cut "In order to", "It should be noted", "As mentioned"

## When reviewing text
If asked to review documentation or copy, check for:
1. Clarity: Can a new team member understand this?
2. Conciseness: Can anything be cut without losing meaning?
3. Accuracy: Do code references match the actual codebase?
4. Structure: Are headings, lists, and formatting used effectively?

Suggest specific rewrites rather than abstract feedback.
